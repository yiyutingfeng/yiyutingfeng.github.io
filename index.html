S3 Glacier Instant Retrieval、S3 Glacier Flexible Retrieval 和 S3 Glacier Deep Archive 存储类专为低成本数据归档而设计。这些存储类提供与 S3 标准和 S3 标准-IA 存储类相同的持久性和弹性。
S3 Glacier Instant Retrieval — 用于归档很少访问且需要毫秒检索的数据。与 S3 Standard-IA 存储类相比,存储在 S3 Glacier Instant Retrieval 存储类中的数据可节省成本,并且具有与 S3 Standard-IA 存储类相同的延迟和吞吐量性能。 S3 Glacier Instant Retrieval 的数据访问成本高于 S3 Standard-IA。
您必须先恢复 S3 Glacier Flexible Retrieval 和 S3 Glacier Deep Archive 对象,然后才能访问它们。 (S3 Standard、RRS、S3 Standard-IA、S3 One Zone-IA、S3 Glacier Instant Retrieval 和 S3 Intelligent-Tiering 对象可随时访问。)
您还可以使用 PUT Object - Copy API 制作对象的副本,将已存储在 Amazon S3 中的对象的存储类更改为任何其他存储类。但是,您不能使用 PUT Object - Copy 来复制存储在 S3 Glacier Flexible Retrieval 或 S3 Glacier Deep Archive 存储类中的对象。您也无法从 S3 One Zone-IA 过渡到 S3 Glacier Instant Retrieval。

数据缓存规则

缓存文件缓存数据规则

  1. 最新的缓存文件file_1不足以存放新数据data1时，缓存文件file_1停止追加新数据，新建缓存文件file_2缓存新数据data1
  2. 缓存容量已达上限，先从缓存文件中删除旧的数据，直到有足够空间存放新的数据，继续向缓存文件追加新数据

可缓存不代表全缓存, 以下情况数据不会被缓存

  1. 待缓存数据长度大于local_data_cache_file_max_size不会被缓存
  2. 受IO线程数影响,受到IO阻塞影响的数据，直接放弃写入缓存文件中,避免因阻塞导致查询耗时倍增
    • 可通过多次查询将因IO阻塞而未被缓存的数据写入到缓存文件中,即逐渐缓存全部可缓存数据的过程,查询会因缓存增加而耗时减少。建议
    • local_data_cache_max_file_num 修改为64,会使得所有IO都用于缓存,可使在第一次查询时缓存所有可缓存数据。不建议,会最阻塞其他SQL执行

缓存文件删除时机

  1. 行云停止或宕机时缓存文件会被删除
  2. 总缓存文件数超过local_data_cache_max_file_num时,旧的缓存文件会被删除

缓存节点选择规则

优先选择已缓存数据占比少的节点,选定节点,该节点会持有10分钟优先权,10分钟后才会重新选择
查询系统表V$LOCAL_DATA_CACHE_STATUS也会触发缓存节点选择

如何判断是否使用数据缓存

  1. 执行节点是可缓存节点
    查看profile中Fragment-后的节点是否在系统表V$LOCAL_DATA_CACHE_STATUS的CACHE_NODES列显示的节点中
